#### canViewSensitivity / allowedSensitivitiesForSession 일관성

- `wiki-sensitivity.ts:45-64` canViewSensitivity: KNOWLEDGE_READ AND [PUBLIC/INTERNAL | RESTRICTED→KNOWLEDGE_REVIEW | SECRET_REF_ONLY→SYSTEM_ACCESS_SECRET]
- `wiki-graph-loader.ts:20-30` allowedSensitivitiesForSession: 4값을 canViewSensitivity로 필터 — 역연산 완전 일치 ✅
- `wiki-graph-loader.ts:128` 2차 방어: DB 필터 후 JS 레이어에서 canViewSensitivity 재확인 — depth-in-defense 준수 ✅

#### shortlist ↔ graph sensitivity 경로

- shortlist: `buildKnowledgeSensitivitySqlFilter` (knowledge_page용) 재타겟팅 (`wpi.sensitivity`) — 스트링 replace 기반 패치. `knowledge_page` 규칙이 `wiki_page_index`와 동일 4값 규약을 공유한다는 전제에서 정상 작동 (W1에서 스키마 규약 확인됨).
- graph: 별도 로직(`allowedSensitivitiesForSession`) 사용 — 두 경로가 다른 함수를 쓰지만 규칙 자체는 동일.
- **관찰 P2:** shortlist는 `buildKnowledgeSensitivitySqlFilter` 공유 헬퍼, graph는 로컬 함수. 추후 W3에서 `buildWikiSensitivitySqlFilter`로 통합하면 중복 제거 가능 — 블로킹 아님.

### 관찰 P1 (경미): `FEATURE_WIKI_FS_MODE`가 feature-flags.ts에 중앙화되지 않음

- `apps/worker/src/jobs/ingest.ts:37-41`에 로컬 `featureWikiFsMode()` 함수 정의:
  ```ts
  function featureWikiFsMode(): boolean {
    return process.env['FEATURE_WIKI_FS_MODE'] === 'true';
  }
  ```
- 해당 주석은 "than churning the shared module"로 의도적 로컬화라고 설명. 기능상 정상 동작.
- 다른 참조 위치(`packages/db/guards/body-column-guard.ts`, 테스트, 문서)도 env 직접 읽기 또는 지역 헬퍼 사용.
- **W3 권장 조치:** 중앙 `feature-flags.ts`로 승격하고 import 통일. 현재는 블로킹 아님 — task 요구사항은 "각 flag가 false일 때 레거시 경로 유지"이며 이는 충족.

## 발견된 이슈

1. **[P2 — 관찰] `FEATURE_WIKI_FS_MODE` 중앙화 누락:** `apps/worker/src/jobs/ingest.ts`에 로컬 정의. 의도적 결정이나 W3에서는 `packages/db/feature-flags.ts`로 옮겨 단일 진실원천 유지 권장.
2. **[P2 — 관찰] sensitivity 필터 헬퍼 이원화:** shortlist는 `buildKnowledgeSensitivitySqlFilter`를 문자열 치환으로 재사용, graph-loader는 로컬 `allowedSensitivitiesForSession`. 둘 다 규약은 일치하지만 W3에서 `buildWikiSensitivitySqlFilter` 공유 헬퍼로 통합하면 drift 예방 가능.
3. **[정상] `review-queue.ts` PII 처리:** 주석(`line 121-124`)에 따르면 `pii` kind가 스키마 enum 주석 목록에 없어서 `sensitivity_promotion`으로 매핑하며 `payload.signal="pii"`로 구분. 스키마 주석과 런타임 동작 정합성 확인 ✅. W3에서 `pii` kind 추가 여부 검토 권장.

## 다음 Phase-W3 준비사항

1. **Feature flag 중앙화:** `FEATURE_WIKI_FS_MODE`를 `packages/db/feature-flags.ts`로 이동, 모든 import 통일.
2. **sensitivity 헬퍼 통합:** `packages/auth/rbac`에 `buildWikiSensitivitySqlFilter` 추가 후 shortlist·graph-loader 양쪽 공유.
3. **review_queue kind enum 정식화:** 주석 대신 pgEnum 또는 CHECK constraint로 강제. `pii` kind 정식 추가.
4. **G8 commit-log 무결성 게이트:** `wiki_commit_log.commitSha == git log HEAD` 런타임 검증 잡 도입 (WIKI-AGENTS §5).
5. **boundary_violation 탐지:** W3-T1 범위. 현재 스키마 주석에 이미 kind가 선언되어 있으므로 구현만 필요.
6. **page-first shortlist 성능 측정:** MAX_NODES=300/topK=20 기준으로 p95 응답시간 벤치마크. 필요시 `trigram` 인덱스 추가 검토.
