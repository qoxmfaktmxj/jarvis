Ask AI 6-lane 라우터 장애 대응

분류 라우터 실패, TPM 초과, 특정 lane이 timeout을 반복할 때의 대응 플레이북. 30분 이내 복구를 목표로 하며, 복구 실패 시 platform-lead 에스컬레이션 경로를 포함합니다.

Ask AI는 6개 lane(wiki-page, wiki-code, policy, case-law, staff, general)으로 질문을 분기합니다. 라우터가 실패하면 전체 질문이 general로 fallback되고 정확도가 급감해요 — 이 런북은 그 상황의 즉시 조치, 진단, 복구, 회복 검증까지 4단계로 설명합니다.

!

On-call이라면 먼저 읽을 것

에스컬레이션 임계값(15분 내 복구 실패 시 platform-lead 즉시 호출)과 고객 공지 템플릿은 섹션 §4에 있습니다.

1. 즉시 조치 (0–5분)

먼저 라우터의 최근 5분 결정 로그를 확인합니다. 로그 출력에 route_failed가 연속 5회 이상 등장하면 분류 모델이 TPM 한도에 걸린 상태이며, 즉시 fallback을 켜야 합니다.

pnpm run logs:router --tail 200 | grep route_failed
# → 연속 5회 이상이면 다음 명령 실행:
pnpm run router:fallback-on --reason tpm-exceeded

fallback 모드는 분류 단계를 건너뛰고 모든 질문을 general로 보내지만, 기본 응답은 유지됩니다. 고객이 정확도 저하를 체감할 수 있으니 상태페이지 공지를 5분 이내 작성하세요.

2. 진단 (5–15분)

원인을 좁히려면 다음 세 그래프를 확인합니다:

  • Grafana → Ask AI Router: lane 별 p50/p95 latency
  • Grafana → OpenAI TPM: 지난 15분 동안 burst가 1.5× 이상 있었는지
  • Worker logs: wiki-agent가 동시에 reindex 중이면 TPM 경쟁 가능

2.1 TPM 초과일 때

일시적 초과면 10분 내 자동 회복됩니다. 15분 이상 지속되면 Organization 설정에서 TPM 조정 티켓을 엽니다. 대기 중엔 fallback 유지.

3. 복구 (15–30분)

fallback을 끄기 전에 아래 3가지를 확인하세요:

  1. 최근 1분 route_failed 0건
  2. p95 latency < 1.2s
  3. TPM 사용률 < 70%
복구 후 30분간 route_accuracy 메트릭을 watch. 이전 baseline 대비 ±5% 이내여야 정상 판단.

4. 회복 검증 & 에스컬레이션

회복 후 Incident 문서를 incidents/YYYY-MM-DD-ask-ai-router.md에 기록하고, 같은 주 금요일 retro에 올립니다.
